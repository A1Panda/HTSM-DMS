# 备份与恢复功能验证清单 (Checklist)

## 后端核心功能验证
- [ ] 成功安装 `multer` 和 `node-cron`。
- [ ] 能否成功生成包含 `version`, `timestamp`, `products`, `codes` 的 JSON 数据结构。
- [ ] 导出备份时，能否正确根据当前的存储模式（MongoDB/文件系统）提取出全量数据。
- [ ] 导入备份时，若为 MongoDB，能否正确处理 ID 映射（生成新的 ObjectId 并更新 Code 表的外键），且原有集合被正确清空。
- [ ] 导入备份时，若为文件系统，能否正确清空 `data/` 目录并正确重建 `products.json` 以及 `*_codes.json` 文件。

## 自动备份功能验证
- [ ] 系统启动时，是否能正确读取并初始化自动备份定时任务。
- [ ] 自动备份是否能在指定时间执行，并在 `data/backups/` 生成合法的 JSON 文件。
- [ ] 备份清理策略是否生效（自动删除超出保留份数的最旧备份）。
- [ ] 自动备份配置接口（获取/更新）是否正常工作，更新后定时任务是否能重新调度。

## 前端功能验证
- [ ] 导航栏是否新增了「数据备份」入口，并能正确跳转至 `/backup` 路由。
- [ ] 手动导出备份按钮能否正确触发浏览器下载行为，且文件后缀、内容合法。
- [ ] 上传备份功能是否能正确限制 `.json` 文件，并在操作前弹出明显的警告确认框。
- [ ] 自动备份配置卡片能否正确加载并保存用户的配置（开关、保留天数）。
- [ ] 服务器备份列表是否能正确拉取并显示，大小和日期格式化是否友好。
- [ ] 列表中的「下载」、「恢复」、「删除」按钮是否均能正确触发对应的后端操作，并有相应的确认和成功提示。

## 极端情况与容错
- [ ] 上传非 JSON 文件或结构损坏的 JSON 文件，系统应有友好的错误提示且不破坏现有数据。
- [ ] `data/backups/` 目录不存在时，系统能否自动创建该目录而不报错。
- [ ] MongoDB 和文件系统模式互相切换导入时，关系型映射（`productId`）没有丢失。
