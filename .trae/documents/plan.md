# 解决超大编码范围导致页面/服务器卡死问题的修复计划

## 摘要 (Summary)
当用户在输入编码起始值和结束值时，如果因为少打或多打了一位导致差值极大（例如 `1` 到 `100000000000`），系统在计算“缺失编码”时会执行全量遍历。由于 JavaScript 单线程的特性，几千万或上百亿次的循环会直接耗尽资源，导致前端浏览器页面完全卡死白屏，同时后端 Node.js 服务器（在执行数据质量统计时）也会被阻塞挂起。
为了彻底、安全地解决此问题，我们将在可能产生长循环的代码处全面引入**“时间 + 迭代次数”的双重熔断机制（Circuit Breaker）**，并在前端表单的输入源头增加范围跨度限制。

## 当前状态分析 (Current State Analysis)
- **`client/src/components/ProductForm.js`**: 表单没有限制起止值的最大差值，允许用户提交超大范围。
- **`client/src/pages/ProductList.js` & `client/src/pages/ProductDetail.js`**: `checkCodeRangeStatus` 函数计算缺失编码时使用了无限制的 `for` 循环。由于不同设备的性能差异，难以给出一个绝对安全的“最大数据量”，一旦范围过大，必将导致 UI 线程卡死。
- **`server/src/controllers/statsController.js`**: `getQualityStats` 接口同样使用了无限制的 `for` 循环。如果有任何一条异常数据，该接口被触发时会挂起整个后端服务。

## 拟定修改 (Proposed Changes)

### 1. 全面引入双重熔断机制 (Circuit Breaker)
在所有涉及 `for (let i = start; i <= end; i++)` 的长循环中，增加熔断保护：
- **前端渲染熔断 (`ProductList.js` & `ProductDetail.js`)**:
  - 在循环开始前记录 `startTime = Date.now()`。
  - 在循环内部每隔一定次数（例如每 1000 次）检查一次时间：`if (Date.now() - startTime > 50)`，即如果该循环执行超过 50 毫秒，则强制 `break` 退出循环。
  - 同时加上最大迭代次数限制（例如 `MAX_ITERATIONS = 50000`），达到该次数也强制退出。
  - 这样能确保无论用户设备性能多差、数据量多大，渲染计算最多只会阻塞 50 毫秒，页面绝对不会卡死。

- **后端统计熔断 (`statsController.js`)**:
  - 在 `getQualityStats` 接口的循环中，同样记录 `startTime`。
  - 限制最大执行时间（例如 `100` 毫秒）或最大迭代次数（例如 `100000` 次），一旦触发熔断条件，立刻 `break`，防止 Node.js 事件循环被阻塞，保障服务器高可用性。

### 2. 限制前端表单输入 (`client/src/components/ProductForm.js`)
- 从源头切断异常数据的产生。
- 在 `codeRanges` 的 `validator` 中增加校验：计算 `end - start` 的值，如果跨度超过 `1000000` (100万)，则阻止提交并提示“号码段范围过大(超过100万)，请分段添加”。

## 假设与决策 (Assumptions & Decisions)
- **决策**: 采用“时间”作为熔断的主要依据，是因为不同用户电脑和服务器的 CPU 性能差异巨大。50 毫秒的卡顿在前端是无感的，这比单纯猜测一个安全的“循环次数”更加科学和稳妥。
- **决策**: 当触发熔断时，我们会停止计算该段范围剩余的缺失编码。此时页面可能无法展示全部的缺失编码，但这是完全可以接受的妥协（因为超大范围本身就是用户的输入错误），最重要的是保证页面能渲染出来，让用户有机会去删除或修改这个错误的产品。

## 验证步骤 (Verification Steps)
1. **表单拦截测试**: 在表单中输入起始值 `1`，结束值 `1000002`，确认无法提交且有明确的范围过大提示。
2. **前端熔断测试**: 模拟数据库中存在一个范围为 `1` 到 `10000000000` 的异常产品。刷新产品列表页和详情页，确认页面能瞬间加载完毕，不会出现长时间白屏或浏览器崩溃。
3. **后端熔断测试**: 在异常数据存在的情况下，调用后端的统计 API（访问仪表盘），确认接口能在极短时间内返回响应，Node.js 服务器正常运行。
