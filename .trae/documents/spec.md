# 备份与恢复功能规范 (Specification)

## 1. 概述
为产品编码管理系统开发一个兼容本地文件存储和 MongoDB 数据库存储的数据备份与恢复功能。在此基础上，新增独立的前端备份管理页面，并实现自动备份功能，确保数据的高可用性和安全性。

## 2. 核心功能
### 2.1 手动备份与恢复
- **导出数据**: 用户可以随时将当前系统的所有产品和编码数据导出为标准的 JSON 文件下载到本地。
- **导入恢复**: 用户可以通过上传符合格式的 JSON 备份文件，覆盖当前系统数据，实现数据恢复。
- **兼容性保障**: 无缝兼容 MongoDB 和本地文件系统两种模式，自动处理 `ObjectId` 与时间戳 `id` 之间的映射与转换。

### 2.2 自动备份
- **定时任务**: 后端通过定时任务（如每天凌晨 2:00）自动执行全量数据备份。
- **本地存储**: 自动备份的 JSON 文件保存在服务器的 `data/backups` 目录下。
- **文件清理**: 支持设置保留天数/份数（如保留最近 7 份），自动清理过期的备份文件。
- **配置管理**: 支持开启/关闭自动备份，设置备份频率和保留份数。

### 2.3 备份管理页面
- **独立页面**: 在前端新增 `/backup` 路由，作为专门的数据备份与恢复控制台。
- **状态面板**: 显示当前存储模式（MongoDB / 文件系统）、数据总量概览。
- **配置区域**: 提供自动备份的开关、频率设置和保留策略配置。
- **历史记录**: 列表展示服务器上已有的自动备份/历史备份文件，提供“下载”、“一键恢复”、“删除”操作。
- **手动操作区**: 提供“导出本地备份”和“上传备份文件恢复”按钮。

## 3. 技术架构设计
### 3.1 依赖与扩展
- **后端**: 
  - 引入 `multer` 处理文件上传。
  - 引入 `node-cron` 用于实现自动备份的定时任务。
- **存储目录**: 
  - 新增 `data/backups` 目录用于存放服务端自动生成的备份文件。
  - 新增 `data/settings.json`（或数据库 Settings 集合）用于持久化自动备份的配置（开启状态、Cron表达式、保留份数）。

### 3.2 后端 API 设计 (`/api/backup`)
- `GET /export`: 导出当前数据，作为附件下载。
- `POST /import`: 接收 `multipart/form-data` 上传，执行数据覆盖恢复。
- `GET /config`: 获取自动备份配置。
- `POST /config`: 更新自动备份配置（并重启/重新调度 cron 任务）。
- `GET /list`: 获取 `data/backups` 目录下的所有备份文件列表（文件名、大小、时间）。
- `GET /download/:filename`: 下载服务器上的指定备份文件。
- `POST /restore/:filename`: 从服务器上的指定备份文件直接恢复数据。
- `DELETE /:filename`: 删除指定的服务器备份文件。

### 3.3 数据兼容性处理 (ID Mapping)
- **MongoDB -> 文件系统**: 导出时，Mongoose 的 `_id` 将被转换为字符串 `id`，文件系统直接将其作为唯一标识，无兼容性问题。
- **文件系统 -> MongoDB**: 导入时，检测 `id` 是否为合法的 `ObjectId`。若不是（如 13 位时间戳），则生成新的 `ObjectId` 并维护新旧 ID 的映射字典，更新所有 `Code` 中的 `productId` 引用，确保关系型外键正确映射。

## 4. 安全与交互体验
- 所有的恢复操作（上传恢复、一键恢复）均为高危操作，前端需弹出明显的二次确认框（Popconfirm/Modal），警告用户现有数据将被完全覆盖。
- 自动备份执行时，如果发生错误，后端需记录详细日志。
